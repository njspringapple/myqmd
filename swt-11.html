Note前几章要点 (Zusammenfassung bisheriger Kapitel)

本章处于软件开发生命周期 (Softwareentwicklungslebenszyklus) 的实现与模块测试 (Implementierung und Modultests) 以及集成与系统测试 (Integration und Systemtests) 阶段。之前的章节涵盖了需求定义 (Anforderungsdefinition)、系统与软件设计 (System- und Softwareentwurf)——包括架构视图、非功能需求影响、架构模式和设计模式。现在我们进入验证和确认的核心环节：测试。

Important本章目标 (Ziel dieser Vorlesung)

在本章结束时，你将理解测试的层次 (Teststufen) 和测试金字塔 (Test-Pyramide)，掌握缺陷成本曲线 (Cost of Defects) 的含义，了解测试驱动开发 (TDD) 的流程，具备设计全面测试用例的能力，并熟悉 Java 和 Python 生态中的主要测试工具。

软件测试 (Testen von Software) 按照测试范围由小到大分为四个层次。模块测试 (Modultests) 也称为单元测试 (Unit Tests)，测试单个模块或函数的正确性。集成测试 (Integrationstests / Komponententests) 测试多个模块之间的协作是否正确。系统测试 (Systemtests) 对整个系统进行端到端的验证。验收测试 (Freigabetests) 从用户或客户的视角验证系统是否满足了预期的需求。

测试的核心目标有两个：快速发现错误 (Fehler schnell finden) 以降低修复成本，以及证明系统的功能是正确的 (Korrekte Funktionalitaet zeigen)。

测试金字塔 (Test-Pyramide) 是理解不同测试层次之间关系的核心模型。金字塔底部是单元测试 (Unit Tests)，数量最多、构建和执行速度最快、隔离性最强，每个测试只关注一个独立的模块或函数。中间层是集成测试 (Integration Tests)，测试多个模块的协作，集成程度更高但构建成本也更大。顶部是系统测试 (System Tests)，覆盖整个系统，集成度最高，但构建和执行最为复杂。

健康的测试策略应当遵循金字塔形状：大量的单元测试构成坚实的基础，适量的集成测试确保模块间的正确协作，少量但关键的系统测试验证端到端的功能。如果金字塔倒置——即系统测试多于单元测试——通常意味着反馈周期过长、维护成本过高。

缺陷成本曲线 (Cost of Defects) 是软件工程中一个经典而重要的概念。它揭示了一个残酷的现实：缺陷被发现得越晚，修复成本就呈指数级增长。在需求阶段 (Requirements) 发现并修复一个缺陷的成本设为基准 1X。如果同一缺陷延迟到设计/架构阶段 (Design/Architecture) 才被发现，成本上升到 3X。到编码阶段 (Coding) 是 7X，到测试阶段 (Testing) 是 15X，而如果直到部署/维护阶段 (Deployment/Maintenance) 才被发现，成本将飙升到 30X 甚至 100X。

这条曲线有力地论证了为什么我们需要早期测试、持续测试，以及为什么测试驱动开发 (TDD) 是一种有效的开发方法——它从编码的最早期就开始验证正确性。

测试驱动开发 (Test-Driven Development, TDD) 又称为”测试优先开发” (Testgetriebene Entwicklung)，其核心理念是先写测试，再写实现。TDD 的循环流程如下：首先识别一个新功能 (neue Funktionalitaet identifizieren)；然后编写一个测试 (Test schreiben)，此时测试必然失败因为功能尚未实现——这就是 “Red” 阶段；接着执行测试 (Test ausfuehren) 确认其确实失败；然后实现功能并进行重构 (Funktionalitaet implementieren und Refactoring durchfuehren)——这就是 “Green” 阶段；再次执行测试，如果测试通过就回到第一步识别下一个功能，如果测试失败则继续修改实现。

TDD 的优势在于：它迫使开发者在编码前思考清楚接口和预期行为；每个功能都有对应的测试保驾护航；重构时有充分的信心不会破坏已有功能。

全面的测试用例设计需要覆盖以下几个维度。对于有效输入：等边三角形 (3,3,3)；等腰三角形的三种排列方式 (3,3,5)、(3,5,3)、(5,3,3)；不等边三角形 (3,4,5)。对于无效输入：边长为零的情况需要对每个参数位置分别测试——(0,3,4)、(3,0,4)、(3,4,0)——以及全部为零 (0,0,0)；负数边长同样需要逐位测试——(-1,3,4)、(3,-1,4)、(3,4,-1)；外观上是等腰但违反三角不等式的情况——(1,1,3)、(1,3,1)、(3,1,1)；外观上是等边但边长为负的情况——(-1,-1,-1)；以及纯粹违反三角不等式的情况——(1,2,4)、(1,4,2)、(4,1,2)。

Tip测试用例设计的关键原则

测试用例设计的核心思想是系统性覆盖：不仅要测试正常情况 (Happy Path)，更要测试边界情况 (Grenzfaelle)、无效输入 (Ungueltige Eingaben)、以及参数排列组合的对称性。每个参数位置都可能隐藏不同的代码路径 (Codepfad)，因此需要分别测试。

单元测试通常遵循 AAA 结构，即 Arrange-Act-Assert 三步法。Arrange (准备) 阶段设置测试所需的对象、数据和前置条件。Act (执行) 阶段调用被测试的方法。Assert (断言) 阶段验证执行结果是否符合预期。每个测试方法应当只测试一个行为 (one behavior per test)，保持测试简洁明确。

此外还有 Setup 和 Teardown 方法：Setup 方法在每个测试之前自动执行，用于创建通用的测试前置条件；Teardown 方法在每个测试之后自动执行，用于清理资源。在 JUnit 中使用 @BeforeEach / @AfterEach 注解，在 pytest 中使用 fixture 机制。

Mock 对象 (Mock-Objekte) 是单元测试中用于替代真实依赖的模拟对象。当被测试的类依赖外部组件——如数据库、网络服务或文件系统——时，直接使用真实组件会导致测试缓慢、不稳定且难以控制。通过使用 Mock 对象，我们可以精确控制依赖的行为，使测试快速、确定性地执行。

在 Java 中，Mockito 是最常用的 Mock 框架。通过 mock(Foo.class) 创建一个模拟对象，然后通过 when(...).thenReturn(...) 指定当特定方法被调用时应返回的值。在 Python 中，标准库的 unittest.mock 模块提供了 MagicMock 类，可以通过设置 return_value 来控制 Mock 的返回行为。

在 Java 生态系统中，JUnit 5 是标准的单元测试框架，提供 @Test、@BeforeEach、@AfterEach 等注解来组织测试。Mockito 是最流行的 Mock 框架，通过 mock() 和 when().thenReturn() 来创建和配置 Mock 对象。Truth (来自 Google) 提供了可读性极强的流畅断言 (Fluent Assertions) API，如 assertThat(notificationText).contains("testuser@google.com")，其替代方案有 AssertJ 和 Hamcrest。jqwik 则支持基于属性的测试 (Property-Based Testing)，自动生成大量测试数据来验证通用属性。

在 Python 生态系统中，pytest 是最受欢迎的测试框架，拥有丰富的插件和强大的 fixture 机制。unittest.mock 是标准库中的 Mock 模块，提供 MagicMock 和 patch() 来模拟依赖。Hypothesis 是 Python 中基于属性的测试库，能够自动生成测试输入并智能地探索边界情况。

课件推荐使用流畅断言 (Fluent Assertions) 而非传统的 assertEquals。流畅断言（如 Truth 的 assertThat(notificationText).contains("testuser@google.com")）读起来更像自然语言，提供更具描述性的错误信息，并且通过链式调用 (Method Chaining) 实现类型安全。传统的 assertEquals(expected, actual) 虽然简单，但参数顺序容易混淆，且错误信息往往不够清晰。
