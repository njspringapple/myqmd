1 编程：艺术还是手艺？ | Programmieren: Kunst oder Handwerk?

Note核心观点 (Kernaussage)

编程不是”艺术 (Kunst)“，而是手艺 (Handwerk)。它需要技巧、纪律和持续的练习——就像木匠或钟表匠的工作。

1.1 代码的意义与任务 | Bedeutung und Aufgabe von Code

同一个任务——对 1 到 10 求和——在不同语言中的表现天差地别。从底层的 x86 汇编到冗长的 COBOL，再到简洁的 Python 一行代码 total = sum(range(1, 11))，语言在进化，但代码本身的核心地位不变。

1.2 KI 时代还需要好代码吗？ | Brauchen wir guten Code trotz KI?

Important回答是：是的！(Ja!)

即使 AI 可以生成代码，人类仍然需要阅读、审查、维护和调试代码。AI 生成的代码质量参差不齐，“整洁代码”的原则反而更加重要——它帮助我们判断 AI 输出的质量，也让代码库在长期维护中保持可控。

2 坏代码的影响 | Der Einfluss von schlechtem Code

2.1 LeBlanc 定律 | Gesetz von LeBlanc

Warning“以后再改” = “永远不改” (Spaeter = Niemals)

这是 LeBlanc 定律的核心。每一次”先这样吧，以后再重构”的决定，几乎都不会真正被兑现。坏代码会不断累积。

n = len(cycle_items)
radius = 3.2
angles = [np.pi / 2 - 2 * np.pi * i / n for i in range(n)]

positions = []
for i, (label, bg_color, edge_color) in enumerate(cycle_items):
    angle = angles[i]
    x = radius * np.cos(angle)
    y = radius * np.sin(angle)
    positions.append((x, y))

    box = mpatches.FancyBboxPatch(
        (x - 1.4, y - 0.7), 2.8, 1.4,
        boxstyle="round,pad=0.15", facecolor=bg_color,
        edgecolor=edge_color, linewidth=2.5
    )
    ax.add_patch(box)
    ax.text(x, y, label, ha='center', va='center',
            fontsize=9, fontweight='bold', color='#333')

# 箭头连接
for i in range(n):
    x1, y1 = positions[i]
    x2, y2 = positions[(i + 1) % n]
    dx = x2 - x1
    dy = y2 - y1
    dist = np.sqrt(dx ** 2 + dy ** 2)
    # 缩短箭头起点和终点
    shrink = 1.5
    sx = x1 + shrink * dx / dist
    sy = y1 + shrink * dy / dist
    ex = x2 - shrink * dx / dist
    ey = y2 - shrink * dy / dist
    ax.annotate('', xy=(ex, ey), xytext=(sx, sy),
                arrowprops=dict(arrowstyle='->', color='#999', lw=2.5,
                                connectionstyle='arc3,rad=0.15'))

plt.tight_layout()

3.2 清晰性：最小惊讶原则 | Klarheit: Principle of Least Astonishment

TipPrinciple of Least Astonishment (最小惊讶原则)

代码的行为应该与读者的预期一致。函数名叫 getUser() 就不应该默默地清除缓存；类名叫 Student 就不应该同时管理课程目录。消除”读代码时的惊讶感”是清晰性的核心目标。

这通过三个手段实现：好的变量命名 (Variablenbenennung)、恰当的注释 (Kommentare)、一致的格式化 (Formatierung)。

4.3 详解：避免错误信息 | Fehlinformationen vermeiden

在强类型语言中，类型信息已经由编译器提供，不需要重复编码到变量名中。例如 Collection<Student> studentList 会误导读者以为底层实现是 List，而实际上可能是 Set。正确做法是直接命名为 students。

在动态类型语言中，如果类型信息对理解至关重要，则应通过名称显式表达，例如 accounts_by_student_name = dict() 比 accounts = dict() 更清晰。

// [x] 坏 (Bad)
Map<String, Account> accounts;
Collection<Student> studentList; // 不一定是 List!

// [+] 好 (Good)
Map<String, Account> accountsByStudentName;
Collection<Student> students;

5 最佳实践 | Best Practices

5.1 类名 | Klassennamen

Note规则 (Regel)

类名应为名词或名词短语 (Substantive oder Substantivphrasen)。技术术语如 Manager, Visitor 等应仅用于对应的技术概念，而非领域概念。Data 或 Info 等词没有实际含义，应避免使用。

// [x] 坏
public record CreateCourseName(String name) {}

// [+] 好
public record CourseName(String name) {}

5.2 方法名 | Methodennamen

Note规则 (Regel)

方法名应为动词或动词短语 (Verben oder Verbphrasen)。

// [x] 坏 — 名词做方法名，语义不清
public void addition(CourseName course, CourseAttendance details) { }
public boolean courseAttendance(CourseName course) { }

// [+] 好 — 动词表达行为
public void add(CourseName course, CourseAttendance details) { }
public boolean didAttendCourse(CourseName course) { }

5.3 领域专用名称 | Domaenenspezifische Namen

使用领域语言 (Domaenensprache) 来命名领域概念，使用技术术语 (Fachbegriffe) 来命名技术概念（算法、数据结构等），不要混淆两者。

// [x] 坏 — 'List' 是技术术语，但这里实现的是 Map
public class AttendanceList implements Map<CourseName, CourseAttendanceData>

// [+] 好 — 名称与实际类型/概念一致
public class AttendanceMap implements Map<CourseName, CourseAttendanceData>
// 或直接
public class Attendances implements Map<CourseName, CourseAttendanceData>

6 注释 | Kommentare

Important核心规则 (Grundregel)

优先改善代码本身，而非添加注释 (Bevorzuge besseren Code statt Kommentaren)。

注释用于表达那些我们无法通过代码充分表达的信息。每个注释都是一个”承认”——承认代码还不够清晰。
