1 为什么需要设计模式 | Warum Entwurfsmuster?

设计模式 (Entwurfsmuster) 不是凭空发明的规则，而是前人在反复遇到相似问题后总结出的经过验证的解决方案模板。它们的核心价值在于三点：让代码对扩展开放、对修改关闭 (Open-Closed Principle)；降低类与类之间的耦合 (Kopplung)；让团队成员通过模式名称就能快速理解设计意图。

下面我们逐一讲解六个最重要的设计模式。每个模式都包含：一个通俗的小故事解释动机，一段”差的代码”展示没有模式时的痛点，一段用模式重构后的代码，对差代码问题的分析，以及该模式的 UML 类图。

2 Strategy 策略模式 | Strategiemuster

2.1 小故事：餐厅结账

想象你开了一家餐厅。一开始只接受现金支付，收银员的工作流程很简单。后来你增加了信用卡支付，收银员就需要在脑子里加一个判断：“如果是现金就数钱，如果是信用卡就刷卡”。再后来又加了手机支付、比特币支付……每增加一种方式，收银员（也就是你的代码）就要多一个 if-else 分支。收银员的”结账手册”越来越厚，每次改一种支付方式的流程，都要翻遍整本手册去修改，而且很容易改错其他无关的部分。

更聪明的做法是：收银员不需要知道每种支付方式的具体操作细节。他只需要说”请处理这笔支付”，然后把具体操作交给一个专门的”支付处理器”。想换支付方式？换一个处理器就行，收银员的工作流程完全不变。

这就是策略模式 (Strategy Pattern) 的核心思想：将变化的算法封装到独立的类中，通过接口统一调用，使得算法可以在运行时自由替换。

2.2 差的代码 | Schlechter Code

// 反面示例: 所有支付逻辑硬编码在一个方法中
public class CashRegister {

    // 每次新增支付方式，必须修改此方法 -- 违反 Open-Closed Principle
    public void processPayment(String type, double amount) {
        if (type.equals("cash")) {
            System.out.println("收到现金: " + amount);
            double change = calculateChange(amount);
            System.out.println("找零: " + change);
        } else if (type.equals("creditcard")) {
            System.out.println("连接银行网关...");
            System.out.println("信用卡扣款: " + amount);
            System.out.println("打印签名单...");
        } else if (type.equals("mobile")) {
            System.out.println("生成二维码...");
            System.out.println("等待手机确认...");
            System.out.println("手机支付成功: " + amount);
        }
        // 如果要加比特币支付？再加一个 else if ...
        // 如果现金流程要改？在这个越来越长的方法里找到对应分支修改
    }

    // 这个方法只跟现金有关，却放在 CashRegister 里
    private double calculateChange(double amount) {
        return amount - 29.99;
    }
}

这段代码有四个核心问题。第一，它违反了 Open-Closed Principle：每新增一种支付方式都必须修改 processPayment 方法，类对修改开放而对扩展关闭，恰好与原则要求相反。第二，它违反了单一职责原则 (SRP)：CashRegister 类同时了解所有支付方式的实现细节，承担了过多职责。第三，各种支付逻辑不可隔离测试。第四，所有逻辑纠缠在一起导致高耦合，修改一种支付方式时可能误改其他分支。

2.4 用 Strategy 模式重构 | Refactoring mit Strategy

// 1. 定义策略接口: 所有支付方式的统一契约
public interface PaymentStrategy {
    void pay(double amount);
}

// 2. 具体策略: 现金支付
public class CashPayment implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("收到现金: " + amount);
        double change = amount - 29.99;
        System.out.println("找零: " + change);
    }
}

// 3. 具体策略: 信用卡支付
public class CreditCardPayment implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("连接银行网关...");
        System.out.println("信用卡扣款: " + amount);
        System.out.println("打印签名单...");
    }
}

// 4. 具体策略: 手机支付
public class MobilePayment implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("生成二维码...");
        System.out.println("等待手机确认...");
        System.out.println("手机支付成功: " + amount);
    }
}

// 5. 上下文: 收银台 -- 不再关心支付细节
public class CashRegister {
    private PaymentStrategy strategy;  // 依赖接口，不依赖具体类

    public CashRegister(PaymentStrategy strategy) {  // Dependency Injection!
        this.strategy = strategy;
    }

    public void setStrategy(PaymentStrategy strategy) {  // 运行时可替换
        this.strategy = strategy;
    }

    public void processPayment(double amount) {
        strategy.pay(amount);  // 委托给策略对象
    }
}

// 使用:
// CashRegister register = new CashRegister(new CashPayment());
// register.processPayment(50.0);
// register.setStrategy(new CreditCardPayment());  // 运行时切换
// register.processPayment(50.0);

新增比特币支付？只需要写一个 BitcoinPayment implements PaymentStrategy，完全不需要修改 CashRegister。

3 Observer 观察者模式 | Beobachtermuster

3.1 小故事：报社与订阅者

想象一家报社。传统做法是每个读者每天早上打电话到报社问：“今天有新报纸吗？”如果一百个读者都这么做，报社的电话线就炸了，而且大部分时候回答都是”还没有”——这就是所谓的轮询 (Polling)，效率极低。

更聪明的做法是：读者只需要到报社登记一次说”我要订阅”。之后每当报社出了新报纸，就主动把报纸送到所有订阅者的信箱里。读者不想看了？打个电话取消订阅就行。

这就是 Observer 模式的核心思想：当一个对象 (Subject/Observable) 的状态发生变化时，自动通知所有依赖它的对象 (Observers)，而不需要观察者反复查询。Subject 和 Observers 之间是松耦合的——Subject 只知道 Observer 接口，不知道具体是谁在监听。

3.2 差的代码 | Schlechter Code

// 反面示例: NewsAgency 直接依赖每一个具体的显示终端
public class NewsAgency {
    private PhoneDisplay phoneDisplay;
    private WebDisplay webDisplay;
    private TVDisplay tvDisplay;

    public NewsAgency(PhoneDisplay phone, WebDisplay web, TVDisplay tv) {
        this.phoneDisplay = phone;
        this.webDisplay = web;
        this.tvDisplay = tv;
    }

    public void publishNews(String news) {
        // 每增加一种显示终端，就要在这里加一行
        phoneDisplay.showOnPhone(news);
        webDisplay.renderOnWeb(news);
        tvDisplay.displayOnTV(news);
        // 新增 TabletDisplay? 修改 NewsAgency!
        // 新增 SmartWatchDisplay? 再修改 NewsAgency!
    }
}

3.4 用 Observer 模式重构 | Refactoring mit Observer

// 1. Observer 接口: 所有观察者的统一契约
public interface NewsObserver {
    void update(String news);
}

// 2. Subject: 新闻社
public class NewsAgency {
    private List<NewsObserver> observers = new ArrayList<>();

    public void subscribe(NewsObserver observer) {
        observers.add(observer);
    }

    public void unsubscribe(NewsObserver observer) {
        observers.remove(observer);
    }

    public void publishNews(String news) {
        // 不需要知道具体是谁在监听!
        for (NewsObserver observer : observers) {
            observer.update(news);
        }
    }
}

// 3. 具体观察者: 手机
public class PhoneDisplay implements NewsObserver {
    @Override
    public void update(String news) {
        System.out.println("[手机推送] " + news);
    }
}

// 4. 具体观察者: 网页
public class WebDisplay implements NewsObserver {
    @Override
    public void update(String news) {
        System.out.println("[网页横幅] " + news);
    }
}

4 Factory Method 工厂方法模式 | Fabrikmethode

4.1 小故事：物流公司的运输工具

你经营一家物流公司，最初只用卡车运货。代码里到处写着 new Truck()。生意红火，客户要求加海运。问题来了：new Truck() 散落在二十个文件里，你要一个一个找出来，判断哪些该改成 new Ship()，哪些该保留——一场灾难。

如果当初你没有在业务代码里直接 new Truck()，而是调用一个”工厂方法”来获取运输工具呢？那么你只需要换一个工厂实现，或者在工厂方法里加一行代码，业务逻辑完全不用动。

Factory Method 模式的核心思想是：将对象的创建过程从使用者中分离出来，由子类或专门的方法来决定创建哪种具体对象。 使用者只依赖于产品接口，不需要知道具体创建了什么。

4.2 差的代码 | Schlechter Code

// 反面示例: 在业务逻辑中直接创建具体对象
public class LogisticsApp {

    public void planDelivery(String type, String cargo) {
        // 业务代码和创建逻辑纠缠在一起
        if (type.equals("road")) {
            Truck truck = new Truck();       // 直接 new 具体类
            truck.loadCargo(cargo);
            truck.deliver();
        } else if (type.equals("sea")) {
            Ship ship = new Ship();          // 又直接 new 另一个具体类
            ship.loadCargo(cargo);
            ship.deliver();
        }
        // 加航空运输? 继续加 else if + new Airplane() ...
        // loadCargo 和 deliver 的调用逻辑完全重复!
    }
}

4.3 为什么差？| Warum ist das schlecht?

这段代码的问题首先在于创建逻辑与业务逻辑混在一起：planDelivery 方法既负责决定创建哪种运输工具，又负责执行装货和运送的业务流程。其次，loadCargo 和 deliver 的调用在每个分支里都重复了——违反 DRY 原则 (Don’t Repeat Yourself)。第三，如果 Truck 的构造函数签名变了（比如需要传入最大载重），你得在所有创建 Truck 的地方修改。第四，无法在测试时替换具体的运输工具。

4.4 用 Factory Method 重构 | Refactoring mit Factory Method

// 1. 产品接口: 所有运输工具的统一契约
public interface Transport {
    void loadCargo(String cargo);
    void deliver();
}

// 2. 具体产品
public class Truck implements Transport {
    @Override
    public void loadCargo(String cargo) {
        System.out.println("卡车装载: " + cargo);
    }
    @Override
    public void deliver() {
        System.out.println("公路运输中...");
    }
}

public class Ship implements Transport {
    @Override
    public void loadCargo(String cargo) {
        System.out.println("轮船装载: " + cargo);
    }
    @Override
    public void deliver() {
        System.out.println("海运中...");
    }
}

// 3. 抽象创建者: 定义工厂方法 + 业务逻辑模板
public abstract class Logistics {

    // 工厂方法: 由子类决定创建什么
    protected abstract Transport createTransport();

    // 业务逻辑: 只依赖 Transport 接口，不重复!
    public void planDelivery(String cargo) {
        Transport transport = createTransport();  // 调用工厂方法
        transport.loadCargo(cargo);
        transport.deliver();
    }
}

// 4. 具体创建者
public class RoadLogistics extends Logistics {
    @Override
    protected Transport createTransport() {
        return new Truck();   // 只在这里决定创建 Truck
    }
}

public class SeaLogistics extends Logistics {
    @Override
    protected Transport createTransport() {
        return new Ship();    // 只在这里决定创建 Ship
    }
}

// 使用:
// Logistics logistics = new RoadLogistics();
// logistics.planDelivery("电子产品");
// 换成海运? Logistics logistics = new SeaLogistics();

5 Singleton 单例模式 | Einzelstueck

5.1 小故事：学校里只有一个校长

想象一所学校。校长只有一位，不管是学生、老师还是家长去找校长，找到的都是同一个人。如果每次有人”请求校长”时都 new Principal() 创建一个新的，那学校就有一百个校长了——谁说了算？谁维护的信息是最新的？完全混乱。

我们需要一种机制：确保一个类在整个程序中只有一个实例，并提供一个全局访问点来获取它。

这就是 Singleton 模式。常见的应用场景包括：配置管理器 (ConfigManager)、日志记录器 (Logger)、数据库连接池、线程池等。

5.2 差的代码 | Schlechter Code

没有 Singleton 模式，任何代码都可以 new ConfigManager()，导致三个问题：多个实例的数据互相不可见，造成数据不一致；每次创建都重复执行昂贵的初始化（读取文件或建立连接），浪费资源；缺少一个统一的全局访问点，导致对象引用在系统中难以管理。

5.4 用 Singleton 重构 | Refactoring mit Singleton

public class ConfigManager {
    // 1. 唯一实例，私有静态字段
    private static ConfigManager instance;

    private Map<String, String> config = new HashMap<>();

    // 2. 私有构造函数 -- 禁止外部 new
    private ConfigManager() {
        loadFromFile("config.properties");
    }

    // 3. 全局访问点 -- 懒加载
    public static ConfigManager getInstance() {
        if (instance == null) {
            instance = new ConfigManager();
        }
        return instance;
    }

    public String get(String key) {
        return config.get(key);
    }

    public void set(String key, String value) {
        config.put(key, value);
    }

    private void loadFromFile(String path) { /* ... */ }
}

// 使用:
// ConfigManager.getInstance().set("db.url", "localhost:5432");
// ConfigManager.getInstance().get("db.url");  --> "localhost:5432" 永远同一个实例!
WarningSingleton 的注意事项

Singleton 经常被过度使用，甚至被称为”反模式”。原因是它本质上是一个全局变量，会引入隐藏的依赖关系，降低可测试性。使用前应确认：这个类是否真的在整个系统中只能有一个实例？如果只是为了”方便获取”，请优先考虑 Dependency Injection。

6 Adapter 适配器模式 | Adaptermuster

6.1 小故事：出国旅行的插头转换器

你带着一台欧式两脚插头的笔记本电脑去英国出差。英国的插座是三脚的，形状完全不同。你不可能把笔记本的电源线剪掉重接（修改现有代码），也不可能要求酒店把所有插座换成欧式的（修改第三方代码）。你需要的是一个转换插头 (Adapter)：一端插入英国插座，另一端提供欧式接口——两边的硬件都不用改。

在软件中，这种情况非常常见：你有一个现有的类（或者第三方库），它的功能完全满足需求，但接口跟你系统期望的不匹配。Adapter 模式将一个类的接口转换为客户端期望的另一个接口，使得原本接口不兼容的类可以一起工作。

6.2 差的代码 | Schlechter Code

// 你的系统期望所有日志器都实现这个接口
public interface Logger {
    void log(String message);
    void error(String message);
}

// 你的业务代码依赖 Logger 接口
public class OrderService {
    private Logger logger;

    public OrderService(Logger logger) {
        this.logger = logger;
    }

    public void placeOrder(String item) {
        logger.log("处理订单: " + item);
        // ... 业务逻辑 ...
        logger.log("订单完成: " + item);
    }
}

// 现在你要接入一个第三方日志库，但它的接口完全不同:
public class FancyThirdPartyLogger {
    public void writeInfo(String text, int level) {
        System.out.println("[INFO-" + level + "] " + text);
    }
    public void writeError(String text, int level) {
        System.out.println("[ERROR-" + level + "] " + text);
    }
}

// 反面做法: 为了使用第三方库，修改 OrderService 的代码
public class OrderService {
    private FancyThirdPartyLogger logger;  // 直接依赖第三方具体类!

    public void placeOrder(String item) {
        logger.writeInfo("处理订单: " + item, 1);  // 方法名变了
        // 所有使用 Logger 的地方都要改...
        // 如果以后换另一个日志库? 再全部改一遍!
    }
}

6.3 为什么差？| Warum ist das schlecht?

核心问题是业务代码直接依赖了第三方库的具体接口。一旦这样做，你的系统中所有使用日志的地方都要从 log() 改成 writeInfo()，从 error() 改成 writeError()。这导致了大量修改、高耦合，并且以后更换日志库时又要全部推倒重来。正确的做法是保持业务代码依赖自己的接口 Logger，通过 Adapter 来桥接。

6.4 用 Adapter 重构 | Refactoring mit Adapter

// 1. 你的 Logger 接口不变
public interface Logger {
    void log(String message);
    void error(String message);
}

// 2. 第三方库不变 (你无法修改它)
public class FancyThirdPartyLogger {
    public void writeInfo(String text, int level) { /* ... */ }
    public void writeError(String text, int level) { /* ... */ }
}

// 3. Adapter: 桥接你的接口和第三方库
public class FancyLoggerAdapter implements Logger {
    private FancyThirdPartyLogger adaptee;  // 持有第三方对象

    public FancyLoggerAdapter(FancyThirdPartyLogger adaptee) {
        this.adaptee = adaptee;
    }

    @Override
    public void log(String message) {
        adaptee.writeInfo(message, 1);  // 转换调用
    }

    @Override
    public void error(String message) {
        adaptee.writeError(message, 3);  // 转换调用
    }
}

7 Decorator 装饰器模式 | Dekorierer

7.1 小故事：咖啡店的加料系统

你走进一家咖啡店。菜单上有基础咖啡 (Espresso, Americano)，还可以加料：加牛奶 (+0.50)、加糖 (+0.20)、加奶油 (+0.70)。每种基础咖啡配上每种加料组合，一共有多少种？Espresso、Espresso加牛奶、Espresso加糖、Espresso加牛奶加糖、Espresso加牛奶加奶油…… 如果每种组合都写一个类 (EspressoWithMilk, EspressoWithMilkAndSugar, ……)，类的数量会爆炸式增长——这就是所谓的类爆炸 (Klassenexplosion)。

Decorator 模式的解决方案是：把”加料”看作是对基础咖啡的”装饰”(包装)。每种加料是一个装饰器，它包裹住一杯咖啡，增强它的功能（增加描述和价格）。你可以像套娃一样一层一层地包装：先用牛奶装饰器包 Espresso，再用奶油装饰器包住牛奶Espresso——组合是无限自由的，却不需要爆炸式的类数量。

Decorator 模式在不修改原有对象的前提下，动态地给对象添加额外职责，比子类继承更加灵活。

7.2 差的代码 | Schlechter Code

核心问题是类爆炸 (Klassenexplosion)：基础种类和装饰选项的组合呈指数级增长。2 种基础咖啡和 3 种加料就产生 16 个类（包含空组合），每新增一种加料，类的数量翻倍。每个类中的价格计算逻辑是重复的 (违反 DRY)，而且所有类之间没有统一的接口。

7.4 用 Decorator 重构 | Refactoring mit Decorator

// 1. 组件接口: 所有饮料（基础和装饰后）的统一契约
public interface Beverage {
    String getDescription();
    double getCost();
}

// 2. 具体组件: 基础咖啡
public class Espresso implements Beverage {
    @Override
    public String getDescription() { return "Espresso"; }
    @Override
    public double getCost() { return 1.99; }
}

public class Americano implements Beverage {
    @Override
    public String getDescription() { return "Americano"; }
    @Override
    public double getCost() { return 2.49; }
}

// 3. 抽象装饰器: 也实现 Beverage，同时持有一个 Beverage
public abstract class CondimentDecorator implements Beverage {
    protected Beverage beverage;  // 被装饰的对象

    public CondimentDecorator(Beverage beverage) {
        this.beverage = beverage;
    }
}

// 4. 具体装饰器
public class MilkDecorator extends CondimentDecorator {
    public MilkDecorator(Beverage beverage) {
        super(beverage);
    }
    @Override
    public String getDescription() {
        return beverage.getDescription() + " + Milk";  // 增强描述
    }
    @Override
    public double getCost() {
        return beverage.getCost() + 0.50;  // 增强价格
    }
}

public class SugarDecorator extends CondimentDecorator {
    public SugarDecorator(Beverage beverage) { super(beverage); }
    @Override
    public String getDescription() { return beverage.getDescription() + " + Sugar"; }
    @Override
    public double getCost() { return beverage.getCost() + 0.20; }
}

public class CreamDecorator extends CondimentDecorator {
    public CreamDecorator(Beverage beverage) { super(beverage); }
    @Override
    public String getDescription() { return beverage.getDescription() + " + Cream"; }
    @Override
    public double getCost() { return beverage.getCost() + 0.70; }
}

// 使用: 像套娃一样自由组合!
// Beverage order = new Espresso();                    // Espresso 1.99
// order = new MilkDecorator(order);                   // + Milk   2.49
// order = new SugarDecorator(order);                  // + Sugar  2.69
// order = new CreamDecorator(order);                  // + Cream  3.39
// System.out.println(order.getDescription());  --> "Espresso + Milk + Sugar + Cream"
// System.out.println(order.getCost());          --> 3.39

总共只需要 2 个基础类 + 1 个抽象装饰器 + 3 个具体装饰器 = 6 个类，可以生成任意组合。新增一种加料只需要新增一个装饰器类。

Tip考试中如何回答设计模式相关题目

回答设计模式题时建议按以下结构组织答案：

(1) 指出问题：当前代码违反了什么原则（OCP? SRP? DRY?）。

(2) 命名模式：明确说出使用的设计模式名称及其类型（行为型/创建型/结构型）。

(3) 画 UML：画出关键的接口、具体类、以及它们之间的关系（implements, extends, 关联）。

(4) 写代码：展示重构后的核心代码，特别是接口定义和关键类。

(5) 说明好处：新代码如何解决了原来的问题（可扩展、可测试、松耦合）。
