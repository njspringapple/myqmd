2 Builder 建造者模式 | Erbauermuster

2.1 小故事：定制电脑的组装过程

你去电脑城配一台电脑。你对店员说：“我要一台 i7 处理器、32GB 内存、1TB SSD、RTX 4090 显卡的电脑。”如果店员的代码是一个巨大的构造函数 new Computer("i7", 32, "1TB", "RTX4090", true, false, true, "水冷", "白色", ...)——十几个参数排成一排，你根本搞不清第三个 true 代表什么，也搞不清哪些参数是可选的。更糟的是，如果某些组件有依赖关系（比如选了水冷必须配特定机箱），构造函数里完全无法体现。

更好的做法是：给你一个”装机清单”(Builder)，你一步步地说”CPU 选 i7”、“内存选 32GB”、“显卡选 RTX 4090”……最后说”组装！“——Builder 负责按照正确的顺序把零件组装成一台完整的电脑。

Builder 模式的核心思想：将一个复杂对象的构建过程与它的表示分离，使得同样的构建过程可以创建不同的表示。 特别适合参数很多、部分可选、或构建步骤有顺序要求的场景。

2.4 用 Builder 重构 | Refactoring mit Builder

// 1. 产品类: 字段设为 final 或 package-private，不暴露 setter
public class Computer {
    private final String cpu;
    private final int ram;
    private final String storage;
    private final String gpu;
    private final boolean hasWifi;
    private final boolean hasBluetooth;
    private final String cooling;
    private final String color;

    // 私有构造函数，只有 Builder 能调用
    private Computer(Builder builder) {
        this.cpu = builder.cpu;
        this.ram = builder.ram;
        this.storage = builder.storage;
        this.gpu = builder.gpu;
        this.hasWifi = builder.hasWifi;
        this.hasBluetooth = builder.hasBluetooth;
        this.cooling = builder.cooling;
        this.color = builder.color;
    }

    @Override
    public String toString() {
        return "Computer[cpu=" + cpu + ", ram=" + ram + "GB, gpu=" + gpu + "]";
    }

    // 2. 静态内部 Builder 类
    public static class Builder {
        // 必填参数
        private final String cpu;
        private final int ram;

        // 可选参数 -- 提供默认值
        private String storage = "256GB";
        private String gpu = "集成显卡";
        private boolean hasWifi = true;
        private boolean hasBluetooth = false;
        private String cooling = "风冷";
        private String color = "黑色";

        // Builder 构造函数只需必填参数
        public Builder(String cpu, int ram) {
            this.cpu = cpu;
            this.ram = ram;
        }

        // 每个 setter 返回 this，支持链式调用
        public Builder storage(String storage) {
            this.storage = storage;
            return this;
        }

        public Builder gpu(String gpu) {
            this.gpu = gpu;
            return this;
        }

        public Builder hasWifi(boolean hasWifi) {
            this.hasWifi = hasWifi;
            return this;
        }

        public Builder hasBluetooth(boolean hasBluetooth) {
            this.hasBluetooth = hasBluetooth;
            return this;
        }

        public Builder cooling(String cooling) {
            this.cooling = cooling;
            return this;
        }

        public Builder color(String color) {
            this.color = color;
            return this;
        }

        // 最终构建方法
        public Computer build() {
            // 可以在这里做校验，比如水冷 + 特定机箱
            return new Computer(this);
        }
    }
}

3 Prototype 原型模式 | Prototypmuster

3.1 小故事：游戏里的怪物复制

你在开发一个游戏。游戏里有一种”哥布林”怪物，它有 50 点生命值、10 点攻击力、穿着皮甲、手持短刀。每当玩家进入一个新房间，需要刷出 20 只一模一样的哥布林。如果每次都 new Goblin() 然后逐个设置属性——setHp(50); setAttack(10); setArmor("皮甲"); setWeapon("短刀");——这不仅繁琐，而且如果哥布林的初始化逻辑需要从数据库或配置文件读取，每次 new 都要重复那些昂贵的操作。

更聪明的做法：先创建一只”模板哥布林” (Prototype)，把它的所有属性设置好。之后每次需要新哥布林时，直接克隆 (clone) 这个模板就行——一行代码搞定，而且不需要重新执行昂贵的初始化。

Prototype 模式的核心思想：通过克隆一个已有的实例来创建新对象，而不是通过 new 从头构建。 尤其适合创建成本高或者需要大量相似对象的场景。

3.2 差的代码 | Schlechter Code

// 反面示例: 每次都从头创建和初始化

3.3 为什么差？| Warum ist das schlecht?

这段代码的问题首先在于重复的初始化逻辑：每次创建新对象都要重复执行相同的属性设置和昂贵的初始化操作（如磁盘读取）。其次，创建逻辑散落各处：想修改哥布林的默认属性，需要在所有 spawn 方法中逐一修改。第三，违反 DRY 原则：普通哥布林和精英哥布林的创建代码高度相似，重复代码过多。第四，如果对象的创建需要经过复杂的计算或外部资源加载，每次从头创建都浪费大量性能。

public interface GameUnitPrototype {

}

// 2. 具体原型
public class Goblin implements GameUnitPrototype {
    private int hp;
    private int attack;
    private String armor;
    private String weapon;
    private byte[] texture;  // 从磁盘加载的纹理，很大

    public Goblin(int hp, int attack, String armor, String weapon) {
        this.hp = hp;
        this.attack = attack;
        this.armor = armor;
        this.weapon = weapon;

5 Proxy 代理模式 | Stellvertretenmuster

5.1 小故事：明星的经纪人

你想找一位大明星拍广告。你不可能直接打电话给明星本人——你得联系他的经纪人 (Proxy)。经纪人跟你谈价格、确认档期、签合同。只有在真正需要拍摄的时候，经纪人才会把明星请来。经纪人起到了几个作用：控制谁能接触明星 (访问控制)、在明星到场前先处理琐事 (延迟加载)、记录所有联系请求 (日志)。

Proxy 模式的核心思想：提供一个代理对象来控制对另一个对象的访问。 代理和真实对象实现同一个接口，客户端不知道自己用的是代理还是真实对象。常见的 Proxy 类型包括：虚拟代理 (Virtueller Proxy，延迟加载)、保护代理 (Schutzproxy，访问控制)、远程代理 (Fernproxy，远程调用)。

        realImage.display();

}

6 Chain of Responsibility 责任链模式 | Zustaendigkeitskette

6.1 小故事：公司的报销审批流程

你在公司提交了一笔报销申请。金额 500 元以内，组长可以直接批准；500 到 5000 元，需要部门经理审批；5000 元以上，需要总经理批准。传统做法是：你写一个大 if-else——如果金额 < 500 找组长，如果 500-5000 找经理，否则找总经理。但是！如果公司增加了一个”副总”审批级别呢？如果审批权限变了呢？每次都要改这段 if-else。

更好的做法是：把每个审批人看作链条上的一环。你只管把申请交给链条的第一环（组长），如果他能批就批了；如果他觉得金额超出自己权限，就自动传递给下一环（经理）——以此类推，直到有人能处理为止。

Chain of Responsibility 模式的核心思想：将多个处理者连成一条链，请求沿链传递，直到有一个处理者能够处理它。 发送者无需知道哪个处理者最终会处理请求。

6.2 差的代码 | Schlechter Code

        // 修改组长的审批上限? 修改这个方法!

6.3 为什么差？| Warum ist das schlecht?

这段代码有三个核心问题。第一，违反 Open-Closed Principle：增加新的审批级别必须修改现有代码。第二，审批逻辑集中在一处：所有审批人的职责和权限都混在同一个方法里，违反 SRP。第三，请求发送者和所有处理者紧耦合：发送者必须知道所有审批级别的规则，无法在运行时动态调整审批链。

6.4 用 Chain of Responsibility 重构 | Refactoring mit Chain of Responsibility

// 1. 抽象处理者
public abstract class Approver {
    protected Approver nextApprover;  // 链中的下一个

    public void setNext(Approver next) {
        this.nextApprover = next;
    }

    public abstract void approve(double amount, String description);

    protected void passToNext(double amount, String description) {
        if (nextApprover != null) {
            nextApprover.approve(amount, description);
        } else {
            System.out.println("没有人能批准此申请: " + description);
        }
    }
}

// 2. 具体处理者: 组长
public class TeamLeader extends Approver {

        if (amount < 500) {
            System.out.println("组长批准: " + description + " (" + amount + "元)");
        } else {
            passToNext(amount, description);  // 超出权限，传给下一个
        }
    }
}

7.1 小故事：图书馆的借阅系统

图书馆有一面书架，书排成一排（数组）。另外还有一个”推荐阅读”列表，用链表存储。管理员想统一遍历这两种数据结构——比如打印所有书名。如果他直接用 for (int i = 0; i < array.length; i++) 遍历数组，用 node = node.next 遍历链表，那他的打印代码就必须知道底层数据结构的细节。换了一种数据结构？打印代码就得重写。

8 State 状态模式 | Zustandsmuster

8.1 小故事：自动售货机的状态转换

想象一台自动售货机。它有几种状态：等待投币、已投币（等待选择商品）、正在出货、缺货。在”等待投币”状态下按选择按钮？什么都不会发生。在”已投币”状态下再投币？应该累加金额或退回。在”正在出货”状态下投币？应该提示等待。

如果用 if-else 来处理：“如果当前状态是等待投币且用户投了币，那么……如果当前状态是已投币且用户按了选择……”——随着状态和事件的增加，if-else 会爆炸式增长，而且状态转换逻辑分散在各个方法中，极易出错。

State 模式的核心思想：将每种状态封装成一个独立的类，把状态相关的行为委托给当前状态对象。 状态转换就是切换当前状态对象，行为随之自动改变——就像售货机换了一个”大脑”。

9 Visitor 访问者模式 | Besuchermuster

9.1 小故事：税务局的上门检查

你是一个城市的税务检查员。城市里有各种建筑：住宅、商店、工厂。对每种建筑，你需要执行不同的检查逻辑——住宅查房产税、商店查营业税、工厂查环保税。如果把检查逻辑写在每种建筑类里面（House.checkTax(), Shop.checkTax()），那么每增加一种检查（比如消防检查），就要修改所有建筑类。

更好的做法是：建筑不需要知道怎么被检查。它只需要说”欢迎检查员来访” (accept(visitor))。检查员 (Visitor) 自己知道对不同类型的建筑该做什么。新增一种检查？写一个新的 Visitor 就行，建筑类不用改。

Visitor 模式的核心思想：将操作与对象结构分离，让你可以在不修改对象类的前提下，定义作用于这些对象的新操作。

// 1. 元素接口: 所有可访问对象的统一契约
public interface Building {

public class House implements Building {
    private double area;

    public void accept(BuildingVisitor visitor) {
        visitor.visitHouse(this);  // 双重分派!

}

public class Shop implements Building {

    public double getRevenue() { return revenue; }

    @Override
    public void accept(BuildingVisitor visitor) {
        visitor.visitShop(this);
    }

    void visitHouse(House house);
    void visitShop(Shop shop);

10 Mediator 中介者模式 | Vermittlermuster

10.1 小故事：机场的塔台

想象一个繁忙的机场。如果每架飞机都要直接跟其他所有飞机通信来协调起降——“我要起飞了，你们让一下”“我在降落，你等一等”——那 10 架飞机就要维护 45 条通信线路 (n*(n-1)/2)。混乱不堪，而且极其危险。

实际上，机场有一个塔台 (Control Tower)。所有飞机只跟塔台通信，由塔台统一协调。飞机 A 对塔台说”我准备好降落了”，塔台通知飞机 B “请在跑道上等待”。飞机之间互相不需要认识。

Mediator 模式的核心思想：用一个中介对象封装一组对象之间的交互，使得对象之间不需要直接引用彼此，从而减少对象之间的耦合。

10.2 差的代码 | Schlechter Code

// 反面示例: 聊天室中每个用户直接持有其他用户的引用
public class User {
    private String name;
    private List<User> contacts = new ArrayList<>();  // 直接认识其他人!

    public User(String name) { this.name = name; }

    public void addContact(User user) { contacts.add(user); }

    public void sendMessage(String message) {
        // 直接发给所有联系人 -- 每个 User 都要维护联系人列表!
        for (User contact : contacts) {
            contact.receive(message, this);
        }
    }

    public void receive(String message, User from) {
        System.out.println(name + " 收到来自 " + from.name + " 的消息: " + message);
    }
}

这段代码中每个 User 都直接依赖其他 User 对象，形成了网状耦合。新增一个用户需要在所有现有用户中 addContact，关系管理是 O(n²) 复杂度。如果需要增加功能（消息过滤、消息记录、权限检查），逻辑会分散到每个 User 类中。

// 1. Mediator 接口
public interface ChatMediator {
    void sendMessage(String message, User sender);
    void addUser(User user);
}

// 2. 具体中介者: 聊天室
public class ChatRoom implements ChatMediator {
    private List<User> users = new ArrayList<>();

    @Override
    public void addUser(User user) {
        users.add(user);
    }

    @Override
    public void sendMessage(String message, User sender) {
        // 中心化的消息分发逻辑
        for (User user : users) {
            if (user != sender) {
                user.receive(message, sender);
            }
        }
    }
}

// 3. 同事类: User -- 只认识 Mediator，不认识其他 User
public class User {
    private String name;
    private ChatMediator mediator;

    public User(String name, ChatMediator mediator) {
        this.name = name;
        this.mediator = mediator;
    }

    public String getName() { return name; }

    public void sendMessage(String message) {
        mediator.sendMessage(message, this);  // 通过中介者发送
    }

    public void receive(String message, User from) {
        System.out.println(name + " 收到来自 " + from.getName() + ": " + message);
    }
}

11 Composite 组合模式 | Kompositum

11.1 小故事：公司的组织架构

一家公司有总经理，总经理下面有几个部门经理，部门经理下面有员工。想计算整个公司的工资总额怎么办？总经理的工资 + 每个部门的工资总额。而”部门的工资总额”= 部门经理的工资 + 该部门所有员工的工资。这是一个树形结构：每个节点可以是一个”叶子”（员工），也可以是一个”容器”（部门，包含子节点）。

如果客户端必须区分”这是一个员工还是一个部门”来决定如何计算工资，代码就会充满 instanceof 和类型判断。Composite 模式让客户端统一对待叶子节点和容器节点——不管是一个人还是一个部门，都调用 getSalary() 就行。

Composite 模式的核心思想：将对象组合成树形结构来表示”部分-整体”的层次结构，使得客户端对单个对象和组合对象的使用具有一致性。

    private String name;
    private double salary;

        this.name = name;
        this.salary = salary;
    }

    @Override
    public String getName() { return name; }

    @Override
    public double getSalary() { return salary; }

    @Override
    public void display(String indent) {
        System.out.println(indent + "员工: " + name + " (" + salary + ")");

// 3. 容器: 部门 -- 也实现相同的接口!
public class Department implements OrganizationComponent {
    private String name;
    private List<OrganizationComponent> children = new ArrayList<>();

    public Department(String name) { this.name = name; }

    public void add(OrganizationComponent component) {
        children.add(component);
    }

    public void remove(OrganizationComponent component) {
        children.remove(component);
    }

    @Override
    public String getName() { return name; }

    @Override
    public double getSalary() {
        double total = 0;
        for (OrganizationComponent child : children) {
            total += child.getSalary();  // 统一调用! 不管是员工还是子部门
        }
        return total;
    }

    @Override
    public void display(String indent) {
        System.out.println(indent + "部门: " + name + " (总计: " + getSalary() + ")");
        for (OrganizationComponent child : children) {
            child.display(indent + "  ");  // 递归显示
        }
    }
}

12 Interpreter 解释器模式 | Interpretermuster

12.1 小故事：计算器的表达式求值

你在写一个简单的计算器。用户输入 "3 + 5" 或 "3 + 5 - 2" 这样的表达式，你需要解析并计算结果。如果用 if-else 来处理每种可能的表达式形式，你会发现组合是无限的。但如果你仔细观察，会发现所有表达式都由基本元素（数字）和运算符（+ - * /）按照固定的语法规则组合而成。

Interpreter 模式的思路是：为语法中的每条规则定义一个类。数字是”终结符表达式”，加法是”非终结符表达式”（它由两个子表达式组合而成）。然后你可以把表达式构建成一棵语法树，调用根节点的 interpret() 就能递归求值。

Interpreter 模式的核心思想：为一种语言定义其文法的表示，并定义一个解释器来解释该语言中的句子。 适合语法简单且效率不是首要关注的场景。

12.2 差的代码 | Schlechter Code

// 反面示例: 硬编码解析逻辑
public class SimpleCalculator {

    public int calculate(String expression) {
        // 手动解析，无法优雅地扩展
        String[] parts = expression.split(" ");
        int result = Integer.parseInt(parts[0]);

        for (int i = 1; i < parts.length; i += 2) {
            String operator = parts[i];
            int operand = Integer.parseInt(parts[i + 1]);

            if (operator.equals("+")) {
                result += operand;
            } else if (operator.equals("-")) {
                result -= operand;
            }
            // 加乘法? 加括号? 加变量? 这个 if-else 会爆炸!
            // 而且不支持嵌套表达式，如 "(3 + 5) * 2"
        }
        return result;
    }
}

12.3 为什么差？| Warum ist das schlecht?

这段代码将解析和计算逻辑全部混在一起，无法处理嵌套表达式和运算符优先级。每增加一种运算符或语法规则，都要修改这个方法。表达式的结构没有被建模为对象，无法复用和组合。

12.4 用 Interpreter 重构 | Refactoring mit Interpreter

// 1. 抽象表达式
public interface Expression {
    int interpret();
}

// 2. 终结符表达式: 数字
public class NumberExpression implements Expression {
    private int number;

    public NumberExpression(int number) {
        this.number = number;
    }

    @Override
    public int interpret() {
        return number;
    }
}

// 3. 非终结符表达式: 加法
public class AddExpression implements Expression {
    private Expression left;
    private Expression right;

    public AddExpression(Expression left, Expression right) {
        this.left = left;
        this.right = right;
    }

    @Override
    public int interpret() {
        return left.interpret() + right.interpret();
    }
}

// 4. 非终结符表达式: 减法
public class SubtractExpression implements Expression {
    private Expression left;
    private Expression right;

    public SubtractExpression(Expression left, Expression right) {
        this.left = left;
        this.right = right;
    }

    @Override
    public int interpret() {
        return left.interpret() - right.interpret();
    }
}

// 5. 非终结符表达式: 乘法 -- 新增运算符无需修改现有类!
public class MultiplyExpression implements Expression {
    private Expression left;
    private Expression right;

    public MultiplyExpression(Expression left, Expression right) {
        this.left = left;
        this.right = right;
    }

    @Override
    public int interpret() {
        return left.interpret() * right.interpret();
    }
}

Tip模式之间的对比 | Vergleich zwischen Mustern

考试中常被问到模式之间的异同：

(1) Strategy vs State：两者结构几乎相同（Context + 接口 + 多个实现类），区别在于 Strategy 是客户端主动选择和替换算法，而 State 是对象自身根据内部状态自动切换行为。

(2) Decorator vs Proxy：两者都包装了另一个对象。Decorator 的目的是增强功能（可以嵌套多层），Proxy 的目的是控制访问（通常只有一层）。

(3) Mediator vs Observer：Observer 是一对多的通知机制（Subject 通知 Observers），Mediator 是多对多的协调机制（Colleagues 通过 Mediator 互相通信）。

(4) Composite vs Decorator：两者都用递归组合。Composite 是”部分-整体”结构，重点在于统一接口；Decorator 是”层层包装”，重点在于动态添加职责。

(5) Factory Method vs Builder：Factory Method 解决的是”创建哪种产品”的问题，Builder 解决的是”如何一步步构建一个复杂产品”的问题。
