1.1 定义：久经考验的解决方案蓝图

Definition: Bewährte Lösungsblaupausen

一个 设计模式（Entwurfsmuster / Design Pattern） 是针对软件开发中 反复出现的设计问题 的一种可复用的解决方案模板。

每个设计模式由四个核心部分组成：

组成部分 / Bestandteil 说明 / Beschreibung
模式名称 (Pattern-Name) 提供通用词汇，方便团队沟通
问题描述 (Problemstellung) 何时使用这个模式？解决什么问题？
解决方案 (Lösung) 类和对象的结构、职责与协作方式
后果 (Konsequenzen) 使用后的优缺点权衡

下面是一个具体的例子——工厂方法（Factory Method）：

部分 内容
名称 Factory Method
问题 需要创建对象，但不想让调用者依赖具体类
方案 用一个工厂方法根据参数返回不同子类的实例
后果 ✅ 解耦具体类 ✅ 易于添加新类型 ❌ 多一层抽象

1.3 热身示例 1：Singleton 模式

Aufwärmbeispiel 1: Singleton-Muster

问题：某些类在整个程序中应该只有一个实例（如数据库连接池、配置管理器）。

解决方案：限制类的实例化，确保全局只有一个实例。

class DatabaseConnection:
    """Singleton 模式：确保全局只有一个数据库连接实例"""
    _instance = None
    
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._connected = False
        return cls._instance
    
    def connect(self):
        self._connected = True
        print("数据库已连接 | Datenbankverbindung hergestellt")
    
    def is_connected(self):
        return self._connected

# 验证：两个变量指向同一个实例
db1 = DatabaseConnection()
db2 = DatabaseConnection()
print(f"db1 is db2: {db1 is db2}")  # True!
db1.connect()
print(f"db2 也已连接？{db2.is_connected()}")  # True!

也可以结合 Factory Method，将单例的存在封装为实现细节：

class Utils:
    """Singleton + Factory Method：隐藏单例细节"""
    _instance = None
    
    def __init__(self):
        raise RuntimeError("请使用 Utils.get_instance()")
    
    @classmethod
    def get_instance(cls):
        """Factory Method: 调用者不需要知道这是单例"""
        if cls._instance is None:
            cls._instance = cls.__new__(cls)
        return cls._instance

# 使用方式：Singleton 对调用者透明
utils = Utils.get_instance()
print(f"获取实例成功: {utils is not None}")

1.4 热身示例 2：Facade 模式

Aufwärmbeispiel 2: Facade-Muster

问题：客户端代码需要直接与多个复杂子系统交互（HTTP 连接、JSON 解析、IO 流……），导致代码紧耦合且冗长。

解决方案：用一个 Facade 类 封装所有子系统的复杂性，提供简洁的接口。

# ====== 没有 Facade：客户端必须了解所有子系统细节 ======
# ====== Ohne Facade: Client muss alle Subsystem-Details kennen ======

#     # 1.1 配置 JSON 库
#     json_parser = JsonParser(strict_mode=True)
#     # 1.2 建立 HTTP 连接
#     conn = HttpConnection("https://api.example.com/user/" + user_id)
#     conn.set_method("GET")
#     conn.set_header("Accept", "application/json")
#     # 1.3 读取响应

#     # → 客户端紧耦合于 JsonParser, HttpConnection, IO 等!

# ====== Mit Facade: Client hängt nur von einer einfachen Schnittstelle ab ======

@dataclass
class User:
    id: str
    name: str
    email: str

@dataclass 
class Reservation:
    id: str
    table_name: str
    user_id: str

class ApiClient:
    """Facade: 封装 HTTP + JSON 等子系统的复杂性"""
    
    def __init__(self, base_url: str):
        self._base_url = base_url
    
    def get_user(self, user_id: str) -> User:
        # 内部处理 HTTP 连接、JSON 解析等细节
        # Intern: HTTP-Verbindung, JSON-Parsing, etc.
        print(f"  [ApiClient] GET {self._base_url}/user/{user_id}")
        return User(id=user_id, name="Alice", email="alice@example.com")
    
    def get_reservation(self, res_id: str) -> Reservation:
        print(f"  [ApiClient] GET {self._base_url}/reservation/{res_id}")
        return Reservation(id=res_id, table_name="Tisch 5", user_id="U1")

Important💡 关键洞察 | Schlüsselerkenntnis

Facade 的本质是信息隐藏（Information Hiding）：客户端只依赖 Facade 的接口，子系统可以随意替换。

但要注意：如果 Facade 承担了过多职责，它可能退化为一个 God Object（上帝对象），这反而违反了单一职责原则。

Facade = Informationsverbergung. Aber Vorsicht: Zu viel Verantwortung → „God Object”!

2.2 解决方案：Builder 模式

Lösung: Builder Pattern

Builder 模式将构建过程分离到一个专门的 Builder 对象中，支持 链式调用（Method Chaining / Fluent Interface）：

from datetime import datetime

class Reservation:
    """餐厅预订类——只能通过 Builder 创建"""
    
    def __init__(self, builder: 'Reservation.Builder'):
        # 私有构造，强制使用 Builder
        self.order_id = builder._order_id
        self.table_name = builder._table_name
        self.group_size = builder._group_size
        self.start_time = builder._start_time
        self.end_time = builder._end_time
        self.notes = builder._notes
        self.needs_wheelchair = builder._needs_wheelchair
        self.prefers_quiet = builder._prefers_quiet
    
    class Builder:
        """链式构建器 | Fluent Builder"""
        
        def __init__(self):
            # 所有字段都有默认值
            self._order_id: str = ""
            self._table_name: str = ""
            self._group_size: int = 1
            self._start_time: Optional[datetime] = None
            self._end_time: Optional[datetime] = None
            self._notes: Optional[str] = None
            self._needs_wheelchair: bool = False
            self._prefers_quiet: bool = False
        
        # 每个 setter 返回 self，实现链式调用
        # Jeder Setter gibt self zurück → Method Chaining
        def order_id(self, val: str) -> 'Reservation.Builder':
            self._order_id = val
            return self
        
        def table_name(self, val: str) -> 'Reservation.Builder':
            self._table_name = val
            return self
        
        def group_size(self, val: int) -> 'Reservation.Builder':
            self._group_size = val
            return self
        
        def start_time(self, val: datetime) -> 'Reservation.Builder':
            self._start_time = val
            return self
        
        def end_time(self, val: datetime) -> 'Reservation.Builder':
            self._end_time = val
            return self
        
        def notes(self, val: str) -> 'Reservation.Builder':
            self._notes = val
            return self
        
        def needs_wheelchair(self, val: bool) -> 'Reservation.Builder':
            self._needs_wheelchair = val
            return self
        
        def prefers_quiet(self, val: bool) -> 'Reservation.Builder':
            self._prefers_quiet = val
            return self
        
        def build(self) -> 'Reservation':
            """最终构建 Reservation 对象"""
            # 可以在此处添加验证逻辑
            if not self._order_id:
                raise ValueError("order_id 不能为空！")
            return Reservation(self)

# ===== Verwendung des Builders (Method Chaining) =====

3.2 解决方案：将状态提取为独立类

Lösung: Zustände als eigene Klassen extrahieren

State 模式的核心思想：让每个状态成为一个类，将状态相关的行为委托给当前状态对象。

class SeatState(ABC):
    """状态接口：定义所有状态共有的行为"""
    
    @abstractmethod
    def reserve(self, seat: 'Seat', order_id: str): ...
    
    @abstractmethod
    def occupy(self, seat: 'Seat'): ...
    
    @abstractmethod
    def release(self, seat: 'Seat'): ...
    
    @abstractmethod
    def display_status(self) -> str: ...


class Available(SeatState):
    """可用状态：可以预订或直接占用"""
    
    def reserve(self, seat: 'Seat', order_id: str):
        seat._order_id = order_id
        seat._state = Reserved()  # 转换到 Reserved 状态
        print(f"  ✅ 座位已预订 (order: {order_id})")
    
    def occupy(self, seat: 'Seat'):
        seat._state = Occupied()  # 转换到 Occupied 状态
        print("  ✅ 座位已占用")
    
    def release(self, seat: 'Seat'):
        raise RuntimeError("座位已空闲，无需释放！")
    
    def display_status(self) -> str:
        return "🟢 可用 | Verfügbar"


class Reserved(SeatState):
    """已预订状态：只能占用"""
    
    def reserve(self, seat: 'Seat', order_id: str):
        raise RuntimeError("座位已被预订！| Bereits reserviert!")
    
    def occupy(self, seat: 'Seat'):
        seat._state = Occupied()
        print("  ✅ 预订者到达，座位已占用")
    
    def release(self, seat: 'Seat'):
        seat._order_id = None
        seat._state = Available()
        print("  ✅ 预订已取消，座位释放")
    
    def display_status(self) -> str:
        return "🟡 已预订 | Reserviert"


class Occupied(SeatState):
    """已占用状态：只能释放"""
    
    def reserve(self, seat: 'Seat', order_id: str):
        raise RuntimeError("座位已被占用！| Bereits belegt!")
    
    def occupy(self, seat: 'Seat'):
        raise RuntimeError("座位已被占用！| Bereits belegt!")
    
    def release(self, seat: 'Seat'):
        seat._order_id = None
        seat._state = Available()
        print("  ✅ 座位已释放")
    
    def display_status(self) -> str:
        return "🔴 已占用 | Belegt"


class Seat:
    """座位类：将所有行为委托给当前状态对象"""
    
    def __init__(self):
        self._state: SeatState = Available()  # 初始状态
        self._order_id: Optional[str] = None
    
    # 行为全部委托给 state！
    # Alles an den State delegieren!
    def reserve(self, order_id: str):
        self._state.reserve(self, order_id)
    
    def occupy(self):
        self._state.occupy(self)
    
    def release(self):
        self._state.release(self)
    
    @property
    def status(self) -> str:
        return self._state.display_status()

# ===== Strategy Interface =====

class FinderStrategy(ABC):
    """策略接口：定义算法契约"""
    
    @abstractmethod
    def find_seats(self, seats: List[SeatInfo]) -> List[SeatInfo]:
        ...

# ===== 具体策略 | Konkrete Strategien =====

class NearestStrategy(FinderStrategy):
    """按距离排序"""
    def find_seats(self, seats: List[SeatInfo]) -> List[SeatInfo]:
        return sorted(seats, key=lambda s: s.distance)

class QuietStrategy(FinderStrategy):
    """筛选安静区域"""
    def find_seats(self, seats: List[SeatInfo]) -> List[SeatInfo]:
        return [s for s in seats if s.is_quiet]

class AccessibleStrategy(FinderStrategy):
    """筛选无障碍座位"""
    def find_seats(self, seats: List[SeatInfo]) -> List[SeatInfo]:
        return [s for s in seats if s.is_accessible]

# ===== Context =====

class SeatFinder:
    """Context：持有一个策略对象，将查找行为委托给它"""
    
    def __init__(self, strategy: FinderStrategy):
        self._strategy = strategy
    
    def find_seats(self, seats: List[SeatInfo]) -> List[SeatInfo]:
        return self._strategy.find_seats(seats)  # 委托！

    SeatInfo("C7", 8.0, True, True),

]

5.1 前置知识：Dynamic Dispatch 的局限

Vorwissen: Grenzen des Dynamic Dispatch

5.3 解决方案：Visitor 模式 + Double Dispatch

Lösung: Visitor Pattern + Double Dispatch

核心思想：将操作从元素类中提取出来，放到独立的 Visitor 类中。 每个 Visitor 代表一种操作。

# ===== Step 1: 定义 Visitor 接口 =====
# ===== Schritt 1: Visitor-Interface definieren =====

class ShapeVisitor(ABC):
    """每种操作实现一个 Visitor"""
    
    @abstractmethod
    def visit_circle(self, circle: 'Circle') -> any: ...
    
    @abstractmethod
    def visit_square(self, square: 'Square') -> any: ...
    
    @abstractmethod
    def visit_rectangle(self, rect: 'Rectangle') -> any: ...

# ===== Step 2: 元素类只需要一个 accept 方法 =====
# ===== Schritt 2: Elementklassen brauchen nur eine accept-Methode =====

class Shape(ABC):
    @abstractmethod
    def accept(self, visitor: ShapeVisitor):
        """Double Dispatch: 自己调用 visitor 的正确方法"""
        ...

class Circle(Shape):
    def __init__(self, radius: float):
        self.radius = radius
    
    def accept(self, visitor: ShapeVisitor):
        return visitor.visit_circle(self)  # 💡 self 的类型是明确的！

class Square(Shape):
    def __init__(self, width: float):
        self.width = width
    
    def accept(self, visitor: ShapeVisitor):
        return visitor.visit_square(self)

class Rectangle(Shape):
    def __init__(self, width: float, height: float):
        self.width = width
        self.height = height
    
    def accept(self, visitor: ShapeVisitor):
        return visitor.visit_rectangle(self)

# ===== Step 3: 每种操作 = 一个 Visitor 类 =====
# ===== Schritt 3: Jede Operation = eine Visitor-Klasse =====

class AreaCalculator(ShapeVisitor):
    """操作1：计算面积"""
    
    def visit_circle(self, circle: Circle):
        return math.pi * circle.radius ** 2
    
    def visit_square(self, square: Square):
        return square.width ** 2
    
    def visit_rectangle(self, rect: Rectangle):
        return rect.width * rect.height

class XmlExporter(ShapeVisitor):
    """操作2：导出 XML"""
    
    def visit_circle(self, circle: Circle):
        return f'<circle radius="{circle.radius}"/>'
    
    def visit_square(self, square: Square):
        return f'<square width="{square.width}"/>'
    
    def visit_rectangle(self, rect: Rectangle):
        return f'<rectangle width="{rect.width}" height="{rect.height}"/>'

# ===== 使用 =====

shapes: List[Shape] = [Circle(5), Square(3), Rectangle(4, 6)]

print("===== 面积计算 (AreaCalculator Visitor) =====")
calculator = AreaCalculator()
for shape in shapes:
    area = shape.accept(calculator)  # Double Dispatch!
    print(f"  {shape.__class__.__name__}: 面积 = {area:.2f}")

print("\n===== XML 导出 (XmlExporter Visitor) =====")
exporter = XmlExporter()
for shape in shapes:
    xml = shape.accept(exporter)
    print(f"  {xml}")

Tip🔍 Double Dispatch 是如何工作的？（点击展开）

让我们追踪一次 circle.accept(calculator) 的调用过程：

  1. circle.accept(calculator) — Python 通过 Single Dispatch 根据 circle 的类型找到 Circle.accept()
  2. 在 Circle.accept() 内部执行 visitor.visit_circle(self) — 此时 self 的类型已经确定是 Circle
  3. 两次 dispatch：第一次根据元素类型选择 accept，第二次根据 visitor 类型选择具体的 visit_xxx

这就是 Double Dispatch —— 运行时行为取决于两个对象的类型。

5.4 现代替代方案：Pattern Matching

Moderne Alternative: Pattern Matching

在 Python 3.10+ 和 Java 17+ 中，Pattern Matching 可以优雅地替代 Visitor 模式中的很多场景：

# ===== Python 3.10+ match/case 替代 Visitor =====
# ===== Python 3.10+ match/case als Visitor-Alternative =====

@dataclass
class CirclePM:
    radius: float

@dataclass
class SquarePM:
    width: float

@dataclass
class RectanglePM:
    width: float
    height: float

ShapePM = CirclePM | SquarePM | RectanglePM  # Union Type

def calculate_area_pm(shape: ShapePM) -> float:
    """无需 Visitor 类！直接 match/case"""
    match shape:
        case CirclePM(radius=r):
            return math.pi * r ** 2
        case SquarePM(width=w):
            return w ** 2
        case RectanglePM(width=w, height=h):
            return w * h

def export_to_xml_pm(shape: ShapePM) -> str:
    """新操作 = 新函数，同样无需修改 Shape 类"""
    match shape:
        case CirclePM(radius=r):
            return f'<circle radius="{r}"/>'
        case SquarePM(width=w):
            return f'<square width="{w}"/>'
        case RectanglePM(width=w, height=h):
            return f'<rectangle width="{w}" height="{h}"/>'

# 使用
shapes_pm = [CirclePM(5), SquarePM(3), RectanglePM(4, 6)]

    print(f"  {s.__class__.__name__}: area={calculate_area_pm(s):.2f}, xml={export_to_xml_pm(s)}")

6.1 为自定义类实现迭代

Iteration für eigene Klassen implementieren

class NumberRange:
    """自定义可迭代类：生成指定范围的数字"""
    
    def __init__(self, start: int, end: int):
        self.start = start
        self.end = end
    
    def __iter__(self):
        """Generator 函数：Python 的 Iterator 语法糖"""
        current = self.start
        while current < self.end:
            yield current     # yield = 暂停并返回值
            current += 1

# 使用 for 循环遍历自定义类

7.3 选择指南：什么时候用什么模式？

Entscheidungshilfe: Wann welches Pattern verwenden?

Tip🧭 模式选择决策树 | Entscheidungsbaum

问题是关于对象创建的吗？ → Builder（参数多）/ Singleton（唯一实例）/ Factory Method（解耦类型）

问题是关于类/对象的组合方式吗？ → Facade（简化接口）/ Composite（树结构）/ Decorator（动态增强）

问题是关于对象之间的交互方式吗？

  • 行为随状态改变？ → State
  • 算法需要可替换？ → Strategy
  • 需要在不修改类的前提下添加操作？ → Visitor (或 Pattern Matching)
  • 需要统一遍历接口？ → Iterator
  • 需要通知多个观察者？ → Observer

最重要的原则：不要为了用设计模式而用设计模式！只有当问题确实存在时，模式才有价值。

Das wichtigste Prinzip: Verwenden Sie Design Patterns nicht um ihrer selbst willen! Patterns sind nur dann wertvoll, wenn das Problem tatsächlich existiert.
